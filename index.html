Getting Started

This year we have built a set of challenges and tutorials to better help new members ease into the workflow of Snowbots.

These challenges are here to get us all up to speed on each others' abilities, experience, and what we still need to learn. Basic assumptions made are that you can clear the first challenge on your own by looking things up online. Otherwise, we do not expect you to have much experience in API design or unit testing coming in, though if you do know all this, great! Regardless of your background, we are here to improve and learn from each other as a group. Debate and questions, especially disagreeing with us if something looks strange, are welcome.


The timeline for this year is:

1. By the first meeting (Sept. 12): Had a go at completing Challenge 1

2. Have Finn check both challenges to be able to sign up for projects

Introduction to the Challenge

The Fizz Buzz

For challenge 1, we thought it we would be boring to just have you solve FizzBuzz or some other simple problem, so to get you comfortable working with existing code, here are a couple of solutions to FizzBuzz, in C++ and Python. We want you to run these and see what they do, then extend them.

Code Samples for the FizzBuzz Challenge

Below are the FizzBuzz challenges in Python and C++ for you to choose from. Pick your choice of language and read through it. The source code can also be downloaded from git by clicking the titles. Your exact tasks will be described underneath the code samples.

Python Sample


import sys

for n in range(1,101):
	fizzbuzz = False
	if n % 3 == 0:
		fizzbuzz = True
		sys.stdout.write('fizz')
	if n % 5 == 0:
		fizzbuzz = True
		sys.stdout.write('buzz')
	if not fizzbuzz:
		sys.stdout.write(str(n))
	sys.stdout.write('\n')

C++ Sample 


#include <iostream>
#include <cstdlib>

int main(){
	for(int i = 0; i < 101; ++i){
		bool fizzbuzz = false;
		if(i % 3 == 0){
			fizzbuzz = true;
			std::cout << "fizz";
		}
		if(i % 5 == 0){
			fizzbuzz = true;
			std::cout << "buzz";
		}
		if(!fizzbuzz){
			std::cout << i;
		}
		std::cout << '\n';
	}
	return EXIT_SUCCESS;
}


References and Resources

A good reference for C++ can be found here: http://www.cplusplus.com/. It has up-to-date, standards-compliant information, and is my go-to 'reference book'. It also has a tutorial, though I'm sure you can find one if that's what you're looking for.

Python has some of the best documentation I have ever seen: https://www.python.org/doc/. This covers from the basics to the deepest technicalities of the language and all its built-in modules.

Interesting libraries/systems we may sometimes use, so you can look up what's in store, and more importantly learn how to use these as we work throughout the year:

Boost

Makes some C++ much easier to read/write. (iterators are awesome – the adventurous may try to rewrite the C++ version below using it)

Armadillo

For that moment when you are tempted to write out the full, gory details of a matrix/other math operation in C++.

CMake

Automates building and configuring software across platforms and computers. We use this for everything.

ROS

Runs our robot, and has a special build system build on top of CMake called Catkin. We also use this for everything.



Challenge 1: Extending FizzBuzz

So, you've picked either C++ or Python, and you got it to compile/run. Now what? We want you to extend its behaviour.

The resulting program (PrimeBuzz) should perform exactly as it did originally, except when it encounters a prime number. Then, it should print “prime”, and only prime, except in the case where it would originally have printed “buzz”. Then, it should prepend “prime” to the original output.

Example Output:

3->Fizz
4->4
23->prime
7->prime
15->FizzBuzz
11->prime
21->fizz
8->8

Bonus points for unit-tested solutions.
Good frameworks for this are:

CATCH for C++

gtest, also for C++

The built-in unittest for Python

Notes on porting to ROS

So, your PrimeBuzz works on the command-line, but what about as a ROS node? Your challenge is to make it run under ROS as a node, with exactly how this happens left to your discretion. Style points are involved, as the point of this is to help us get to know each other as programmers, and to see how we cope with the kind of things we will have to do throughout the year.

The main thing we have in mind is that your design should be reasonably flexible, readable and future-proof. If you do not feel like you could come back to your code in a month and fix some obscure bug, either you can try refactoring it so it is more readable, or if there's something strange or unintuitive you need to do for it to work, commenting your rationale on the line above said thing can work too.

If you have a question or get stuck, come find me [Finn] at the first session and we can go from there. This challenge is designed to extend into at least the second session, and is a catalyst for open debate, not a test.

Unit Testing

Bonus points for unit tests are still a thing for challenge 2 – try having a look at http://wiki.ros.org/rostest. We hope to keep this practice going for the entire project. If you remember CPSC 110 too well, then maybe the later stages of this project can help you see unit/integration/regression tests for how helpful they really are. Not having them can cause several types of pain in longer-term projects. (ask me [Finn] for examples if you like)
